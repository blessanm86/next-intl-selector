Engineering proposal · 2026-04

The TypeScript ceiling on our message catalog, and how we get past it without losing DX.

Our en.json is the source of AsMessageKey — a TypeScript union of every translation key. As the catalog has grown, that union has gotten expensive enough to start failing the type checker. This proposal swaps the literal-string API for a property-selector API, validated against the production code, and shows the perf and ergonomics numbers that came out the other side.

01What's actually broken

A type-system limit, surfacing as deps-array errors and as string casts.

AsMessageKey is a TypeScript type. It's the union of every leaf in our en.json — roughly 7,300 string literals. That's what makes t("Some.Key") reject typos at compile time.

The trouble is what TypeScript does with that union when it shows up in a complex position. Two of these in a tuple? It computes the cross product. Three? Worse. Four? You hit error TS2590: Expression produces a union type that is too complex to represent. 

// gcp-resource-detail-page.tsx — pre-port
const tabs = useMemo(() => [...], [
  config.tabLabels.attributes as string,  // ← workaround
  config.tabLabels.overview   as string,  // ← workaround
  config.tabLabels.telemetry  as string,  // ← workaround
]);
/* eslint-disable react-hooks/use-memo, react-hooks/exhaustive-deps */

PR #11751 patched around it with as string casts and an eslint-disable block. Two callsites, three workarounds, one comment apologizing for it. That's the visible shape of the problem today.

And it only gets worse: every new key grows the union, and every callsite that places AsMessageKey in a tuple or deps array pays a cost that scales with the union size. The workarounds survive only as long as nobody adds a key that lands in a hot type position.

Why now? We migrated typecheck to tsgo (TypeScript 7 beta), which makes TS2590 deterministic instead of order-dependent. The bug went from "sometimes fires" to "always fires."

02The fix — functions, not strings

Pick the leaf with property access. TypeScript checks the chain instead of the union.

Same translation, two ways:

Before — string literal
const t = useTranslations();
t("MainNavigation.items.home")
// ↑ TS validates against AsMessageKey,
// the 7,300-member literal union.
After — selector function
const t = useMessageT();
t(m => m.MainNavigation.items.home)
// ↑ TS validates each property access
// structurally. No union materialised.

Why it's faster: TypeScript walks m.MainNavigation against the IntlMessages shape, then .items against the inner shape, then .home. Each step is an indexed access. There's no giant union, no cross product, no tuple expansion. Translations render byte-for-byte identical at runtime.

Prior art. i18next's TypeScript Selector API — same pattern, same motivation, different runtime. We've also proposed this upstream to next-intl with a reproduction repo and benchmarks.

Go-to-definition fix. Cmd+Click on m.MainNavigation now jumps straight to en.json. next-intl's docs recommend declare interface IntlMessages extends Messages {} — an interface with declaration merging support we don't use. Flattening to type IntlMessages = typeof import("en.json") removes a hop from the type chain and lets TypeScript navigate directly to the JSON property.

//                       ^^^^^ Property 'itemms' does not exist

//                                is not assignable to 'string'

Editor experience: better, actually

Today: typing t(" opens a flat 7,300-item completion list. Heavy. With selectors: typing m. shows the ~150 top-level groups; each . narrows to the local subtree. The catalog browses like a tree instead of pasting like a wall.

ICU and rich tags work unchanged

t.rich(m => m.Metrics.Sidebar.bucketBoundaries.showMore, {
  count: remaining,                  // ICU plural
  em: chunks => <em>{chunks}</em>,    // rich tag
});
// → "5 <em>more boundaries</em>"

Selectors as prop types

type Props = { label: MessageSelector };

// At the call site:
<Tab label={m => m.Hub.title} />

MessageSelector is a function type — multiple of these in a tuple don't trip TS2590. That's how gcp-resource-detail-page.tsx drops its three workarounds.

04The numbers

How much faster does the type checker run after the port? Two views: the worst-offender file, and the whole project.

View 1 — the file the type checker hated most

use-get-column-label.ts is one line of code that took 8.7 seconds by itself for the type checker to evaluate. That single line was comparing a runtime value against the 7,300-member AsMessageKey union. After the port, that file drops out of the hot list entirely — the work just isn't there anymore.

The framing detail. 8 files ported — 0.08% of the project — bought a 14.8% project-wide speedup. The remaining 319 callsites are individually small, but the full migration also removes every remaining as AsMessageKey cast and latent TS2590 risk. Realistic full-port total: around 20–25% from baseline.

05Decisions we still need

One question the audit already answered.

  1. Files where the translation key isn't a literal string at the call site audited

    43 production sites use as AsMessageKey casts. Audit breakdown:

    BucketSitesTreatment
    Restructurable — path is statically constructible, just built across a boundary ~33 Refactor to pass selectors. ~21 are the PageLayoutAction button family — one pre-flight PR.
    Genuinely runtime — key only exists at runtime 4 Escape hatch: t.translateRaw(path), lint-restricted.
    Test/storybook fixtures 11 Trivial codemod or leave on legacy hook.

    Recommendation: ~90% restructure / ~10% escape hatch.

Out of scope: removing AsMessageKey entirely, cleaning up StringLiteral<T> consumers, modularising en.json. Each is a separate initiative.

06Migration: one big-bang PR

327 callsites, one PR, no coexistence period.

  1. ESLint auto-fix rule in eslint-plugin-dash0

    Rewrites t("MainNavigation.items.home") → t(m => m.MainNavigation.items.home) mechanically. The rule is the source of truth for migration semantics — reviewers read that, not 327 individual diffs.

  2. Run the codemod + manually handle edge cases

    Run pnpm lint:fix across the codebase. Hand-port the ~33 restructurable dynamic-key sites (mostly the PageLayoutAction button family) and the ~4 genuinely-runtime sites that need hasLeafRaw.

  3. Remove the legacy surface in the same PR

    8 symbols across 2 files. No coexistence period — clean cut. Audit below; full inventory in legacy-cleanup-audit.md.
